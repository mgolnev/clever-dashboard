# 0003. Режим сравнения периодов

- Статус: Принято
- Дата: 2026-06-12

## Контекст

Руководство запросило возможность сравнивать произвольный период с **любым**
вторым периодом (день↔день, неделя↔неделя, месяц↔месяц, год↔год), а не только с
автоматически вычисляемым предыдущим. До этого сравнение было неявным и жёстким:
backend всегда считал «предыдущий период той же длины непосредственно перед
текущим» и отдавал `current`+`prev`, а frontend рисовал дельты. Выбрать второй
период было нельзя; сравнение присутствовало только на части вкладок.

Требование «режим» трактуем как **глобальный сквозной фильтр**: один тумблер
включает/выключает сравнение и применяется ко всем вкладкам дашборда.

## Решение

1. **Опциональный явный период сравнения на backend.** Эндпоинты
   `/api/metrics`, `/api/funnel`, `/api/logistics` принимают необязательные
   query-параметры `compareStart`/`compareEnd` (`YYYY-MM-DD`). Если они пусты —
   поведение прежнее (предыдущий период той же длины). Это даёт **полную
   обратную совместимость**.
2. **Хелпер `resolvePrevRange`** добавлен в каждый из трёх доменных пакетов
   (`metrics`, `logistics`, `funnel`) — атомарность сохранена, домены не
   импортируют друг друга ради общей логики. (Сознательное дублирование вместо
   общего пакета — см. вопросы ниже.)
3. **Воронка получила сравнение.** В ответ `/api/funnel` добавлены `previous`
   (Range) и `prevStages` ([]Stage). Сегменты/проблемы за второй период не
   считаются — вне scope.
4. **Глобальный state режима на frontend** (`App.tsx`): `compareEnabled`,
   `compareMode` (`prev`|`yoy`|`prevMonth`|`custom`), `compareStart`/`compareEnd`.
   Второй период для пресетов вычисляется на клиенте и уходит в API явно;
   дельты (`delta()` + `DeltaBadge`) при этом не меняются.
5. **UI в `DateRangeBar`** — тумблер «Режим сравнения», пресеты и поля
   произвольного второго периода. Дефолт: режим включён + `prev` (сохраняет
   прежнее поведение).
6. **Гейтинг и overlay.** Проп `showCompare` скрывает дельты при выключенном
   режиме. Наложение двух периодов добавлено на воронке (`FunnelChart`) и в
   динамике (`DynamicsTab`, режим «без разреза»).
7. **Корректное форматирование дат** на клиенте через `formatLocalDate` (без
   `toISOString`), чтобы в зонах восточнее UTC (UTC+3) не было сдвига дат на день.

## Последствия

- `/api/dynamics` (grouped breakdown) и сегменты воронки сравнение **не**
  поддерживают — overlay динамики работает только для одиночной серии.
- Логика расчёта второго периода продублирована в трёх пакетах.
- Дельты остаются вычислением frontend; backend отдаёт абсолюты обоих периодов.

## Вопросы, к которым надо вернуться

1. **Календарное выравнивание серий.** В `custom`-режиме overlay динамики и
   сравнение серий сопоставляются по индексу слева (форма динамики), а не по
   календарным датам. Для периодов разной длины и для YoY может потребоваться
   выравнивание «дата к дате» (1-е число к 1-му).
2. **Общий хелпер дат на backend.** `resolvePrevRange` дублируется в `metrics`/
   `logistics`/`funnel`. Если правил станет больше (напр. сложные пресеты), стоит
   выделить нейтральный пакет `daterange`, не нарушая независимость доменов.
3. **Сравнение на остальных срезах.** `/api/dynamics` grouped breakdown и
   сегменты/проблемы воронки за второй период пока без сравнения — расширить при
   запросе.
4. **Пресеты YoY/месяц и длина месяца/високосный год.** Сдвиг на год/месяц
   делается календарно (`setFullYear`/`setMonth`); для 29 февраля и месяцев
   разной длины поведение JS-Date нормализует дату — проверить, что это
   устраивает бизнес.
5. **Сохранение состояния режима.** URL-синхронизация (`?compareStart=…`) и
   шеринг ссылкой осознанно отложены — вернуться, если потребуется делиться
   срезами.
6. **Производительность.** Явный второй период = второй набор запросов к БД
   (как и сейчас для prev). При больших произвольных диапазонах оценить нагрузку.
