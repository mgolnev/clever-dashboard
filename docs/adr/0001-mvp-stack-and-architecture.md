# 0001. Стек, коннекторное ядро и модель данных MVP-дашборда

- Статус: Принято
- Дата: 2026-05-30

## Контекст

Нужен дашборд для еженедельного обзора интернет-магазина CleverWear.ru. На первом
этапе источник — Битрикс (выгрузка заказов в файле); далее планируются Яндекс
Метрика и Аппметрика. Требования согласованы с заказчиком:

- загрузка данных Битрикса **через файл** (ручная) на старте;
- товарная аналитика нужна сразу;
- выбор произвольного периода в UI и сравнение с **предыдущим периодом той же длины**.

## Решение

1. **Стек:** Go + Fiber (backend), React + TypeScript + Tailwind (frontend),
   согласно конвенциям `AGENTS.md`.
2. **БД:** SQLite по умолчанию (запуск без внешних зависимостей), Postgres через
   env. Переносимый SQL, ветвление через `db.IsPostgres()`.
3. **Коннекторное ядро:** приём данных изолирован в модуле `ingestion`; источник
   (файл сейчас, Bitrix API/Метрика/Аппметрика позже) — сменный вход, не влияющий
   на витрину и метрики.
4. **Атомарные сервисы:** `ingestion`, `normalize`, `orders`, `metrics` —
   независимые модули; связь через нейтральный `model` и общую БД.
5. **Модель данных:** `raw_import` (журнал загрузок) → `orders` (PK = номер
   заказа, дедуп) → `order_items` (позиции из столбца «Позиции»).
6. **Источник позиций:** столбец «Позиции» (`[offer_id] Название (N шт)`) + цены
   из «Цена товара» по порядку. Соответствие 1:1 проверено на полной выгрузке
   (1328 заказов → 4823 позиции, 0 расхождений) — отдельная выгрузка «по позициям»
   не требуется.
7. **Сравнение периодов:** предыдущий период = примыкающий слева интервал той же
   длины.

## Последствия

- Быстрый старт без инфраструктуры (SQLite-файл).
- Подключение Bitrix API / Метрики / Аппметрики не потребует переделки витрины.
- Атрибуты товара (бренд/категория/пол/размер) выводятся эвристикой из названия;
  при необходимости точности — обогащение справочником по `offer_id` (см. ROADMAP).
