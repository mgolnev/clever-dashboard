# Инструкции для AI-агентов (GJ Auto Merch)

> **Обязательно для агента:** этот файл — источник истины по процессу разработки. Cursor подключает его через `.cursor/rules/development-workflow.mdc` (`alwaysApply: true`). В начале задачи с изменением кода **прочитай AGENTS.md**; в конце — выполни чеклист ниже, не описывай перезапуск словами вместо действия.

Документ задаёт поведение агента при доработках репозитория. Человек-разработчик может опираться на те же правила.

## Обязательный чеклист после доработок

**Не считай задачу выполненной**, пока не сделано всё по списку:

| # | Шаг | Когда | Действие агента |
|---|-----|--------|-----------------|
| 1 | Тесты | Всегда после правок кода | `go test ./... -count=1`; при `frontend/` — `cd frontend && npx tsc --noEmit` |
| 2 | **Перезапуск backend** | Любые правки Go: `cmd/`, `internal/`, `go.mod` | Выполни в Shell (см. [Перезапуск сервисов](#перезапуск-сервисов-обязательно)) |
| 3 | **Перезапуск frontend** | Любые правки `frontend/src/`, `package.json`, конфигов CRA | Выполни в Shell (см. ниже) |
| 4 | Отчёт пользователю | Всегда | Явно укажи: тесты OK, backend `:8080` перезапущен, frontend `:3000` перезапущен (или что не трогалось) |

Перезапуск **не опционален**: формулировки «если сервисы уже запущены» / «пользователь может сам перезапустить» **запрещены**. Агент сам останавливает старый процесс и поднимает новый через терминал.

### Перезапуск сервисов (обязательно)

1. Проверь папку **terminals** в сессии Cursor — есть ли уже `make run`, `go run`, `npm start`.
2. Освободи порт и запусти заново:

```bash
# Backend (из корня репозитория) — в фоне, block_until_ms: 0
./scripts/restart-dev.sh backend
# или вручную:
# lsof -ti:8080 | xargs kill -9 2>/dev/null; make run

# Frontend — в фоне, block_until_ms: 0
./scripts/restart-dev.sh frontend
# или: lsof -ti:3000 | xargs kill -9 2>/dev/null; cd frontend && npm start
```

3. Hot reload **не заменяет** перезапуск:
   - `make dev` (Air) — достаточно для Go, если Air уже запущен агентом в этой сессии;
   - без Air после правок Go — **обязателен** перезапуск backend;
   - после правок API-контрактов, env, `package.json` — **обязателен** перезапуск frontend, даже если CRA обычно hot-reload.

| Сервис | Порт | Команда запуска |
|--------|------|-----------------|
| Backend | 8080 | `make run` или `./scripts/restart-dev.sh backend` |
| Backend (watch) | 8080 | `make dev` (Air) |
| Frontend dev | 3000 | `cd frontend && npm start` (proxy → 8080) |

## Атомарность и независимость сервисов

Каждый разрабатываемый **сервис** (доменный модуль) должен быть **атомарным** и **независимым**. Это обязательное архитектурное требование при проектировании и правках кода.

### Атомарность

- **Одна ответственность** на сервис — одна причина для изменения (поиск, похожие, скоринг, каталог, стратегия, импорт фида и т.д. — разные модули).
- **Go:** бизнес-логика в `internal/services/<domain>/` или `internal/discovery/<search|similar|strategy>/`; handler только принимает HTTP, валидирует вход и вызывает сервис.
- **Не создавай «комбайны»** — сервис, который одновременно меняет поиск, скоринг и каталог, нарушает правило; разбей на отдельные пакеты/типы.
- **Frontend:** доменная логика в отдельных `hooks/` и вызовах API; страница не должна напрямую дублировать чужую бизнес-логику.

### Независимость

- Сервис **не импортирует внутренности** соседнего домена (пакеты `search` ↔ `similar` ↔ `catalog` не вызывают друг друга напрямую).
- Связь только через **явные контракты**: интерфейсы репозиториев, DTO, HTTP API, общие типы в нейтральном пакете — не через «удобный» импорт чужого `Service`.
- **DI** (`internal/container/`) собирает зависимости; сервис получает порты, а не конкретные реализации чужого слоя.
- Изменение одного сервиса **не должно требовать** правок в несвязанных пакетах (кроме wiring в container/routes и документации API).
- **Тесты** пишутся на уровне пакета сервиса (`go test ./internal/discovery/search/...`) без поднятия всего приложения.

### Чеклист агента перед merge-готовым кодом

1. Новая логика лежит в **своём** пакете/модуле, а не размазана по чужому handler/service?
2. Нет циклических импортов и скрытых зависимостей между доменами?
3. Контракт API (эндпоинт, типы ответа) описан/обновлён в `docs/*-api.md`, если менялся внешний интерфейс?
4. При сомнении «расширить существующий сервис» vs «выделить новый» — **предпочитай новый атомарный сервис** и ADR, если решение архитектурное.

Подробнее о слоях: [docs/architecture.md](docs/architecture.md), [docs/architecture-client.md](docs/architecture-client.md).

## Роли и уровень

### Frontend (`frontend/`)

Веди разработку как **senior React-разработчик**:

- TypeScript, функциональные компоненты, хуки; без лишних абстракций.
- Следуй существующей структуре: `pages/`, `components/`, `hooks/`, `services/api.ts`.
- Стили — Tailwind; витрина Shop и PDP — [docs/gj-design-master-prompt.md](docs/gj-design-master-prompt.md) и `.cursor/rules/gj-design-master-prompt.mdc`.
- Перед правками UI смотри соседние компоненты и не ломай контракты API.
- Доменные фичи — отдельные hooks/модули; см. [атомарность сервисов](#атомарность-и-независимость-сервисов).

### Backend (Go, корень репозитория)

Веди разработку как **senior Go-разработчик**:

- Слои: handlers → services → repository; SQL в `internal/queries/`, DI в `internal/container/`.
- Соблюдай [атомарность и независимость сервисов](#атомарность-и-независимость-сервисов): не смешивай домены в одном `Service`.
- Не обходи существующие паттерны (Fiber, `db.IsPostgres()` для ветвления БД).
- Публичные контракты API — обратная совместимость, если не оговорено иначе в ADR.
- Комментарии — только для неочевидной бизнес-логики.

## Архитектурные решения (ADR)

Перед выбором подхода **прочитай** [docs/adr/](docs/adr/) — принятые ADR обязательны к учёту.

После значимого решения **зафиксируй его в ADR**, если это уместно по [docs/adr/README.md](docs/adr/README.md):

- новый слой, контракт API, схема БД;
- зависимость с заметным влиянием;
- труднообратимое решение (движок поиска, миграция);
- согласование бэкенд / фронт / админка.

Без ADR: косметика, точечные баг-фиксы, узкие фичи без архитектурного следствия.

Формат: `docs/adr/NNNN-kebab-title.md`, обновить таблицу в `docs/adr/README.md`.

## Анализ данных по заказам (обязательно к прочтению)

**Перед любой задачей анализа данных по заказам** (метрики, воронки, новые срезы,
интерпретация полей выгрузки Битрикса) сначала прочитай:

1. [docs/bitrix-fields.md](docs/bitrix-fields.md) — **словарь всех 117 полей**
   выгрузки: за что отвечает каждое поле, тип, заполненность, связи (ключи,
   параллельные массивы позиций, зависимость адресов от службы доставки). Не
   угадывай смысл поля — сверяйся со словарём.
2. [docs/funnel-analysis.md](docs/funnel-analysis.md) — модель пути заказа,
   маппинг статусов в стадии воронки и ключевые аналитические выводы.
3. [docs/import-bitrix.md](docs/import-bitrix.md) — контракт импорта и разбор
   столбца «Позиции».

При появлении новых полей/срезов — обнови `docs/bitrix-fields.md`.
