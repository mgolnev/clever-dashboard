# Анализ пути заказа и модель воронки

Документ описывает, как из данных Битрикса строится воронка и какие выводы она
помогает увидеть. Цифры ниже — по полной выгрузке (1328 заказов, сен 2025 – май
2026); в сервисе всё считается за выбранный период.

## Жизненный цикл заказа

Гросс-заказ (создан) → оплата онлайн → сборка/обработка → отправка → доставка в
ПВЗ → выкуп (Выполнен). На каждом шаге возможен отвал: неоплата → авто-закрытие,
отмена, проблема на сборке, невыкуп, возврат.

### Модель оплаты: только предоплата (проверено по данным)

Анализ выгрузки (1328 заказов, май 2026) подтвердил: **постоплаты / оплаты при
получении в магазине нет** — модель 100% онлайн-предоплатная.

- У всех 1328 заказов заполнена `Платежная система`, и все методы предоплатные:
  Картой, СБП, Яндекс Пэй / Яндекс Пэй Сплит, «Оплата по реквизитам **(предоплата)**».
- Признаков постоплаты (`наложенный платёж`, `при получении`, `наличные`,
  `курьеру`, `постоплата`) нет ни в `Платежная система`, ни в `Служба доставки`,
  ни в `Комментариях`, ни в полях доставки. `Тип плательщика` — везде физлицо.
- Поэтому `Оплачен = Нет` (332 заказа) трактуется как **брошенная онлайн-оплата**
  (305 из них в статусе «Закрыт»), а не как легитимная оплата при получении.
  Логика воронки (`is_paid` как этап «оплачено») корректна.
- Аномалия для контроля: 3 заказа в статусе «Выполнен» с `Оплачен = Нет` — в
  предоплатной модели не должно быть; вероятно ручное закрытие/возврат.

### Маппинг статусов Битрикса в стадии

| Статус | Стадия | Трактовка |
|--------|--------|-----------|
| Не подтвержден | `new` | создан, ждёт оплаты |
| Оплачен / Принят / Обработан / Ожидает отправку / Собирается | `processing` | оплачен/в обработке/в сборке |
| Отправлен | `shipped` | отгружен |
| Прибыл в ПВЗ | `in_pvz` | доставлен, ждёт выкупа |
| Выполнен | `completed` | выкуплен (успех) |
| Закрыт (неоплачен) | `closed`/`canceled` | брошенная оплата / отмена |
| Возврат заказа / частичный возврат / **полный возврат средств** | `returned` | возврат после доставки |

> Маппинг проверен по данным (2026-05): после правок не осталось статусов в
> `other`. Ранее «Совершён возврат средств» ошибочно попадал в `other` (не
> учитывался в возвратах/терминальных статусах), а буквальный статус «Оплачен» —
> тоже в `other`; исправлено на `returned` и `processing` соответственно.

Стадии воронки **кумулятивны**: заказ учитывается на стадии, если дошёл хотя бы
до неё (напр. «Выполнен» уже прошёл «Отправлен»).

> **Этап «Оплачен» в воронке** засчитывается, если `is_paid = true` **ИЛИ** заказ
> продвинулся дальше оплаты (статус `processing/shipped/in_pvz/completed/returned`).
> Причина: ~16 заказов находятся в сборке/выполнены с незаполненным флагом
> `is_paid`; без этого правила стадия «Оплачен» оказывалась бы меньше «В сборке»,
> и кумулятивная воронка теряла монотонность. KPI «Оплачено» на вкладке «Обзор»
> по-прежнему считает строго по флагу `is_paid` (фактическая онлайн-оплата).

## Ключевые выводы из данных

1. **Главный отвал — оплата онлайн.** Гросс 1328 → оплачено 996 (**75%**), т.е.
   ~25% (≈305 заказов) закрываются неоплаченными. Поле «Причина отмены» почти не
   заполняется (98% пусто) → это брошенная онлайн-оплата, а не явные причины.
   - **Уточнение по полю `Оплата ПС`** (богаче флага `Оплачен`): из 332
     неоплаченных **320 — «Не проводилась»** (клиент даже не начал оплату) и лишь
     **12 — «Не успешна»** (платёж пытались провести, но он не прошёл). Связи
     идеальные: «Успешна» → оплачен в 100% (886), «Не проводилась» → не оплачен в
     100% (320). Вывод: основная утечка — это **отказ начать оплату**, а не
     технические сбои платежа; работать нужно с моментом перехода к оплате.
   - **`Доставка разреш.` ≈ гейт по оплате:** «Нет» у 334 заказов ≈ 320 из них не
     оплачены; «Да» у 982 оплаченных. Доставка фактически разрешается после оплаты.
   - **`Статус (просмотр)`** содержит, кто менял статус: `import` (авто) или
     фамилия менеджера. Большинство «Закрыт» — авто-закрытие неоплаченных (`import`).
2. **Способ оплаты — главный фактор отмены:**
   - Яндекс Пэй Сплит — оплата **57.7%**, отмена **41.8%** (хуже всех по объёму);
   - Яндекс Пэй — оплата 64.5%, отмена 35.5%;
   - Картой — оплата 75.9%, отмена 22.6%;
   - **СБП — самый здоровый: оплата 81.4%, отмена 18.6%**.
3. **Канал app/сайт не влияет** на оплачиваемость (~75% у обоих) — проблема не в
   платформе, а в платёжном сценарии.
4. **Доставка:** Яндекс ПВЗ лучший по оплате (83.6%), Почта России и самовывоз
   ~72%, СДЭК 68%.
5. **Сборка/отгрузка:** 33 проблемных заказа — нехватка остатка
   («Недостаточное количество товара …») и ошибки фискализации чека.
6. **Выкуп:** из отправленных до «Выполнен» доходит ~86% (часть свежих заказов
   ещё в пути — стадии после «Отправлен» частично отражают незавершённость).

## Что с этим делать (гипотезы)

- Разобрать сценарий оплаты Яндекс Сплит/Пэй (таймауты, ретраи, напоминания о
  неоплате) — это крупнейший управляемый рычаг выручки.
- Подсветить СБП как приоритетный способ оплаты.
- Контроль остатков перед публикацией (проблемы «недостаточное количество»).
- Мониторинг ошибок фискализации чеков.

## Разрезы воронки в сервисе

Вкладка «Воронки» строит кумулятивную воронку и разрезы по: **способу оплаты,
службе доставки, каналу, региону** — с метриками оплата% / отмена% / выкуп% и
подсветкой проблемных сегментов, плюс топ проблем сборки и причин отмены.
